# LlamaCppProvider — FFI approach (design sketch)

> **Status:** design, not built. This is the execution plan for the embedded
> runtime deferred from `juice_llm` 0.1.0 (SPEC §"0.1.0 as built", impl #2). It
> is the real "on-device, no server process" story — distinct from the example's
> `OllamaLlmProvider`, which talks to a separate `ollama serve`.
>
> Grounded 2026-06-11 against the current ecosystem; exact llama.cpp symbol
> names must be regenerated by ffigen against a pinned commit (the C API churns).

## The decision in one line

**Wrap the existing `llama_cpp_dart` binding in a companion package
`juice_llm_llamacpp`, rather than hand-rolling ffigen or putting native code in
`juice_llm` core.** Fall back to hand-rolled ffigen + build hooks only if the
wrapper proves too constraining.

### Why a companion package, not core

`juice_llm` core is dependency-free and seam-only on purpose (the `FlagsSource`
convention). A runtime that drags in a native toolchain, Metal shaders, and a
multi-MB `.dylib`/`.so` must not become a transitive dependency of every app
that uses the bloc. So the `LlamaCppProvider` lives outside core — and because
it carries native build assets and its own platform matrix, it deserves its own
package (`juice_llm_llamacpp`) rather than living as an example recipe like
`OllamaLlmProvider` does. **This is a ROADMAP catalog addition — Kevin's call
before it's minted** (the catalog is locked; cf. decision #6 reasoning).

### Why wrap `llama_cpp_dart`, not hand-roll

| | Wrap `llama_cpp_dart` (0.9.x) | Hand-roll ffigen + build hooks |
|---|---|---|
| Native build | the binding owns it | we own `hook/build.dart` + `native_toolchain_c` |
| Isolate/threading | **managed isolate included** | we build it |
| API churn | absorbed by the binding | we regenerate ffigen each bump |
| Control | bounded by its API | total |
| Time-to-Phase-A | days | weeks |

The binding is at 0.9.x ("one more breaking pass before 1.0") with iOS/Android/
macOS support and a Flutter-friendly managed isolate — exactly the hard parts.
Wrapping it gets Phase A working fast; we keep the hand-rolled path documented as
the escape hatch if its API can't express cancellation/streaming the way the
seam needs. ([llama_cpp_dart](https://pub.dev/packages/llama_cpp_dart) ·
[build hooks stable in 3.38](https://docs.flutter.dev/platform-integration/bind-native-code))

## The shape that matters: the provider is isolate-owned

This is the single most important design fact, and it's *already* a clean fit
for our bloc.

llama.cpp objects (`llama_model`, `llama_context`, samplers) are raw pointers
that **cannot cross a Dart isolate boundary**, and `llama_decode` is blocking
GPU/CPU work that **must not run on the UI isolate**. Therefore a *worker
isolate owns the entire model lifecycle* — load, generate, embed, unload — and
the `LlamaCppProvider` on the main isolate is a thin facade that marshals
messages to it and streams token pieces back over a `SendPort`/`ReceivePort`.

`llama_cpp_dart`'s managed isolate gives us this; if hand-rolling, we build a
long-lived isolate with a command protocol (`Load`, `Generate`, `Stop`,
`Embed`, `Unload`).

Crucially, **this maps onto `LlmBloc` with no bloc changes**: `beginGeneration`
already wraps `provider.generate(request)` as a stream and finalizes through one
`GenerationOutcome`; the throttle, the `sequential` queue, and the cancel path
are all runtime-agnostic. The provider just has to honor the seam contract:
*stream chunks; stop the runtime when the listener cancels.*

## Seam mapping

| `LlmProvider` method | llama.cpp / isolate action |
|---|---|
| `load(modelPath, opts)` | spawn worker; `llama_model_load_from_file(modelPath)`; `llama_init_from_model` with `n_ctx = opts.contextTokens`, `n_gpu_layers = opts.gpuLayers`, `n_threads = opts.threads`. Throws `LlmProviderException` on null model (bad GGUF) or OOM. |
| `generate(request)` | on the worker: apply Gemma chat template to `request.messages` → tokenize → prime KV cache via `llama_decode` → **loop**: `llama_sampler_sample` → `llama_vocab_is_eog`? stop : detokenize the piece → send `LlmChunk(piece)` to main → append token, `llama_decode`. Stop also on `params.maxTokens` / a `stop` string. Honor `params.temperature`/`topP` via the sampler chain. |
| cancel (listener cancels the stream) | `generate()`'s `async*` `finally` sends `Stop` to the worker; the worker checks a stop-flag between decode steps and returns. KV cache is cleared (`llama_memory_clear`) so the next generate starts clean. |
| `embed(text)` | a context created with `embeddings = true` (or a second pooled context); `llama_decode` then read `llama_get_embeddings_seq`; return the vector. Requires `LlmCapability.embeddings` (already guarded in the bloc). |
| `unload()` / `dispose()` | free sampler/context/model on the worker; `dispose` also kills the isolate. |
| `capabilities` | `{text}` always; `{embeddings}` if built/loaded for it; `{vision}` only with an mmproj projector (Gemma 4 multimodal — Phase D). |

### Provider skeleton (sketch — not compiled; lives in `juice_llm_llamacpp`)

```dart
class LlamaCppProvider implements LlmProvider {
  LlamaCppProvider({this.capabilities = const {LlmCapability.text}});
  @override final Set<LlmCapability> capabilities;
  @override String get name => 'llama_cpp';

  _Worker? _worker; // owns the isolate + the llama.cpp pointers

  @override
  Future<void> load(String modelPath, LlmLoadOptions o) async {
    _worker = await _Worker.spawn();
    final ok = await _worker!.command(LoadCmd(modelPath, o));
    if (!ok) throw const LlmProviderException('failed to load model (bad GGUF / OOM)');
  }

  @override
  Stream<LlmChunk> generate(LlmRequest request) async* {
    final w = _worker;
    if (w == null) throw const LlmProviderException('generate() with no model loaded');
    final rx = w.startGeneration(request); // ReceivePort of token pieces
    try {
      await for (final msg in rx) {
        if (msg is _Token) yield LlmChunk(msg.piece, tokens: 1, done: msg.eog);
        if (msg is _GenError) throw LlmProviderException(msg.message);
        if (msg is _Token && msg.eog) break;
      }
    } finally {
      w.stopGeneration(); // out-of-band stop → worker clears KV cache
    }
  }

  @override
  Future<List<double>> embed(String text) async {
    if (!capabilities.contains(LlmCapability.embeddings)) {
      throw UnsupportedError('embeddings not enabled');
    }
    return _worker!.command(EmbedCmd(text));
  }

  @override Future<void> unload() async => _worker?.command(const UnloadCmd());
  @override Future<void> dispose() async { await _worker?.kill(); _worker = null; }
}
```

The bloc-side flow is unchanged: this `generate()` plugs straight into
`beginGeneration`; the `finally`-driven `stopGeneration()` is precisely what the
existing cancel path expects.

## macOS specifics (Phase A target)

- **Backend:** build llama.cpp with `GGML_METAL=ON`. `n_gpu_layers` (from
  `LlmLoadOptions.gpuLayers`) offloads to the GPU; default high = "as many as
  fit."
- **Metal shaders:** llama.cpp needs its Metal library at runtime. Either embed
  shaders (`GGML_METAL_EMBED_LIBRARY=ON`, simplest — no `default.metallib` to
  locate) or bundle the `.metallib` as a resource. Embedding is the path of
  least friction for a Flutter macOS bundle.
- **Build/bundling:** prefer **build hooks** (`hook/build.dart` +
  `package:native_toolchain_c`, stable since 3.38) to compile/fetch the native
  lib and bundle it as a code asset — this replaces the old "prebuilt binary in
  the plugin" dance. `llama_cpp_dart` may still use the classic plugin model;
  whichever the wrapper picks, the companion package owns it, not core.
- **Signing/entitlements:** we already sign with the personal-team dev cert
  (Glean's pbxproj). A bundled `.dylib` must be signed too — Flutter's macOS
  build handles framework signing, but verify the native asset is signed under
  the same identity. No new entitlement is needed for Metal/CPU inference
  (it's not network/sandbox-sensitive); the keychain entitlement we already
  carry is unrelated.
- **Binary size:** the llama.cpp lib adds a few MB to the app; model *weights*
  are separate and acquired via `ModelSource` (never bundled).

## Gemma 4 specifics

- **Weights:** GGUF, quantized (QAT Q4 variants keep E2B in a ~few-GB on-disk
  footprint). Acquired via a real `ModelSource` (HTTP Range resume + streaming
  SHA-256 — the verify contract the seam already specifies); Apache-2.0, so no
  license-acceptance gate.
- **Chat template:** Gemma uses `<start_of_turn>user … <end_of_turn>
  <start_of_turn>model`. Apply the model's built-in template via
  `llama_chat_apply_template` rather than hand-formatting — our `LlmMessage`
  list (`system`/`user`/`assistant`) maps directly onto its message array.
- **Multimodal (Phase D):** Gemma 4 E2B is natively multimodal; vision needs the
  mmproj projector loaded alongside the text weights and image tokens spliced
  into the prompt. Defer until text (A) + embeddings (B) + RAG (C) land; the
  `vision` capability + `LlmMessage.images` are already in the seam for it.

## Risks / decisions for Kevin

1. **Companion-package mint** — `juice_llm_llamacpp` is a new catalog entry in a
   locked roadmap. Approve before building, or start it as an example recipe and
   promote it once proven (lower-ceremony first cut).
2. **Wrap vs hand-roll** — recommendation is wrap `llama_cpp_dart`. Risk: its API
   may not expose mid-stream cancellation cleanly; first execution step is a
   spike that proves cancel + streaming through its managed isolate before
   committing.
3. **Build-hook maturity in practice** — stable as of 3.38, but the first real
   native-asset build on this machine is itself a checkpoint; budget time for it.
4. **Pin llama.cpp** — pick a commit, pin it, regenerate ffigen against it; treat
   bumps as deliberate (the C API moves).

## Phase A execution checklist

1. Spike: a throwaway Flutter macOS app that loads a small GGUF via
   `llama_cpp_dart`, streams tokens off-isolate, and **cancels mid-stream**.
   This de-risks the two unknowns (build + cancel) before any package work.
2. Decide companion-package vs example-recipe (Kevin) → scaffold it.
3. Implement `LlamaCppProvider` against the seam (skeleton above); reuse the
   exact `FakeLlmProvider` contract tests by running them against a tiny real
   GGUF behind a local-only flag (never CI).
4. Real `ModelSource` (HTTP resume + streaming SHA-256) — replaces
   `FileModelSource` for the fetch path; the package's verify contract is
   already specified.
5. Wire into Glean: the Almanac's nightly note synthesizes the day's own
   entries — zero recall surface — proving the whole stack end-to-end.
